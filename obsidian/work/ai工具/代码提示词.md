# AI Code 规范

## 基本原则

- 先不写代码。
- 尽量复用已有代码。
- 不兼容之前的代码。
- 说清楚怎么改、结果是什么、还有什么风险和问题。
- 不写代码，只给落地方案。
- 不清楚的问题需要先看代码以及表格没有答案再问我

## 代码要求

- 类声明直接使用类名，不要写包名加类名，统一直接导入包。
- 每个方法不能超过 50 行。
- 代码要简洁，职责要清晰，尽量复用。
- 协议名称、字段名称遵循项目规范。
- 消息名称用项目规范，小驼峰，不用下划线。
- 数据库字段如果有对象映射，写库时拼成 `String`，读库时转成类类型，转换方法尽量放在一起。

## 注释和日志

- 除了变量声明和赋值，其他都要写注释。
- 方法、方法块、参数都要写注释。
- 方法块超过 3 行的非声明变量也要写注释。
- 所有流程走不下去的地方都要打印日志。
- 出口返回和错误分支都要带详细信息。
- 代码块也要加注释。

## 协作方式

- 任何决定不要自己做，先问我。
- 需要看代码和相关配置时，先看代码，再用 MCP 看配置。
- 不知道配置时先问我。
- 每次都把需要改动、需要新增的协议、DAO 和代码说清楚。

## 工具偏好

- 尽量都使用 MCP 工具。
- 想需求、对齐思路时用 `/grill-with-docs`。
- 准备动手时用 `/caveman`。
- 遇到复杂 Bug 时用 `/diagnose`。
- 写核心纯函数或底层业务逻辑时用 `/tdd`。
- 代码如果只用一次，就不要抽重。
